## Channel-by-channel

### 01 — IG seed-graph traversal
- **What ran:** `python discover_ig_graph.py --fetch --limit 5` then `--aggregate`
- **Cost:** Apify ~$0.02/seed × 5 = $0.10
- **Result:** 5 seeds → 100 posts/seed (where available) → **112 candidate venues**
- **Quality:** mixed — `taggedAccounts` recovers a noisy mix:
  - Real restaurant handles: `restaurantdaniel`, `lebernardinny`, `odetterestaurant`
  - Personal chef handles: `danielboulud`, `ericripert`, `chefjoseandres`
  - Media accounts: `michelinguide`, `forbestravelguide`, `laliste1000`
  - Photographers, family members, food brands
- **Limitations:**
  - Aggregator emits raw IG handles, not resolved venue names. Need a second pass:
    look up each handle's IG bio → extract "headquartered in {city, state}" → join
    to the canonical row.
  - `locationName` field is sparsely populated (chefs tag venues via tagged users
    more often than via geo-location).
  - 50 seeds is below the 150-target. Backfill JBF semifinalists, GuildSomm
    Advanced cohort, and Eater 38 chefs (planned ~100 additional seeds).
- **Follow-up:**
  - Add an IG-handle → venue resolver: Apify IG profile-scraper on each
    unique handle, then keep only those whose bio mentions a US address.
  - Scale to full 50-seed run after the resolver lands.

### 02 — Restaurant job boards
- **What ran:** all 5 sources sequentially
- **Cost:** ~150 Serper queries/source × $0.001 = ~$0.75 total
- **Result:**
  - Indeed (via Serper): **497 rows** ✅
  - Sevenrooms Hire: **8 rows** ✅
  - Restaurant Zone: 0
  - Poached: 0 (page structure changed; module fetches the wrong endpoint)

- **Quality concern:** the Indeed snippet parser is too permissive — many rows
  capture the *location* string ("Atlanta, GA 30303") or *job title* ("Sommelier
  Blossom") as the employer name instead of the actual restaurant. Need a
  post-process pass that drops rows where `name` contains `, [A-Z]{2}\s*\d{5}`
  or is a known job-title fragment.
- **Follow-up:**
  - Add an Indeed-output cleanup pass (`scripts/clean_jobs_indeed.py`).
  - Re-investigate Poached: their site moved to a new domain or restructured.
  - For Restaurant Zone: the Serper-snippet path returns links but no employer
    in the snippet. Switch to direct fetch + Apify-actor mode.

### 04 — D2C marketplaces
- **What ran:** `python discover_directories.py --source d2c_goldbelly|d2c_murrays|
  d2c_williams_sonoma|d2c_zingermans`
- **Result:** all four returned 0 rows.
- **Why:**
  - Goldbelly, Murray's, Williams-Sonoma all sit behind Cloudflare/Akamai 403s
    against requests + UA. The `strategy="llm"` HTTP-then-Playwright fallback
    rendered the page but the merchant cards load via lazy JS that
    `playwright.networkidle` doesn't reliably wait for.
  - Zingerman's "food tours / our-partners" page is curated tours, not the
    cheese/charcuterie partner list we wanted. Bad URL choice.
- **Follow-up:**
  - For Goldbelly: switch to JSON-API capture inside Playwright (the merchant
    grid is a `/_next/data/.../merchants.json` payload).
  - For Williams-Sonoma: drop — the "marketplace" experiment shut down in 2023.
    Reuse the budget for Mouth or Eat Boutique.
  - For Murray's: scrape `/our-cheesemakers` via Playwright with explicit
    `wait_for_selector('.cheesemaker-card')`.
  - For Zingerman's: switch URL to `zingermansdeli.com/about/sources/` or drop.

## Master union

`scripts/build_wave2_master.py` collects all per-source CSVs across the 8
channels, dedupes by (name, city, state), and stacks contributing sources
per venue. Run after all channels complete:

```bash
python scripts/build_wave2_master.py --stamp 20260525
```

Output: `output/wave2_master_<YYYYMMDD>.csv` with `channel`, `n_sources`,
`sources`, `channels`, `distinctions` per row.

A row with `n_sources >= 2` means the venue was found via multiple Wave 2
lenses (e.g., on the CMS sommelier list AND in a Vittles writeup AND in the
Heritage Foods customer list) — these are the highest-conviction leads in
Wave 2.
