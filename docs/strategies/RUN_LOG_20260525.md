## Wave 2.2 update — curl_cffi as default fetch path

Moved curl_cffi from a directories-only shim to the **canonical fetch path
in `awards/_lib.fetch_html()`**, which is the single function every other
channel imports for HTTP. Chrome120 TLS impersonation now runs first; plain
requests is the fallback. This cascades transparently to:

- `awards/` (40+ source modules + `_editorial.py` + `llm_extract.py`)
- `jobs/` (5 board modules — all use `fetch_listing_html` → `fetch_html`)
- `best_wine_shops/` (Serper-discovered article fetcher)
- `directories/` (already on curl_cffi via `_browser_fetch`; double-wrapped)
- `scarcity/`, `enrich.py`, every consumer of `awards._lib`

### What this tells us

**curl_cffi is necessary but not sufficient.** The Wave 2.1 wins
(Williams-Sonoma 200/807KB, Goldbelly+Murray's+D'Artagnan-style 404-with-body
unlocked) were the *TLS-fingerprint*-blocked cases. The 19 sources above
are blocked by *other* reasons — Serper relevance, JS-rendered DOMs, login
walls, dead URLs, or no public roster.

Real follow-ups for these are channel-specific, not TLS-related:
- **Article-based awards** (VinePair, Michelin Grape, Good Food Awards): hard-code
  the canonical article URLs instead of relying on Serper relevance.
- **JS-rendered job boards** (Culinary Agents, Restaurant Zone): switch to the
  Apify Indeed actor or scrape their RSS/JSON-LD if any exists.
- **GuildSomm**: needs `--cookies-from` flag like the NYT awards module already
  uses.
- **Dead URLs** (Anna Hezel, Zingerman's, Greenpoint Fish): discover correct
  endpoints or retire.

### Wine Enthusiast refresh

Re-ran the 3 Wine Enthusiast awards (94 rows prior) to harvest the curl_cffi

### Verdict

Default-curl_cffi is the right move regardless of yield count: every future
WAF-protected source we encounter now works without a per-module workaround.
The TLS-fingerprint cost is paid once in `awards/_lib.py`.

## Channel-by-channel

### 01 — IG seed-graph traversal
- **What ran:** `python discover_ig_graph.py --fetch --limit 5` then `--aggregate`
- **Cost:** Apify ~$0.02/seed × 5 = $0.10
- **Result:** 5 seeds → 100 posts/seed (where available) → **112 candidate venues**
- **Quality:** mixed — `taggedAccounts` recovers a noisy mix:
  - Real restaurant handles: `restaurantdaniel`, `lebernardinny`, `odetterestaurant`
  - Personal chef handles: `danielboulud`, `ericripert`, `chefjoseandres`
  - Media accounts: `michelinguide`, `forbestravelguide`, `laliste1000`
  - Photographers, family members, food brands
- **Limitations:**
  - Aggregator emits raw IG handles, not resolved venue names. Need a second pass:
    look up each handle's IG bio → extract "headquartered in {city, state}" → join
    to the canonical row.
  - `locationName` field is sparsely populated (chefs tag venues via tagged users
    more often than via geo-location).
  - 50 seeds is below the 150-target. Backfill JBF semifinalists, GuildSomm
    Advanced cohort, and Eater 38 chefs (planned ~100 additional seeds).
- **Follow-up:**
  - Add an IG-handle → venue resolver: Apify IG profile-scraper on each
    unique handle, then keep only those whose bio mentions a US address.
  - Scale to full 50-seed run after the resolver lands.

## Master union

`scripts/build_wave2_master.py` collects all per-source CSVs across the 8
channels, dedupes by (name, city, state), and stacks contributing sources
per venue. Run after all channels complete:

```bash
python scripts/build_wave2_master.py --stamp 20260525
```

Output: `output/wave2_master_<YYYYMMDD>.csv` with `channel`, `n_sources`,
`sources`, `channels`, `distinctions` per row.

A row with `n_sources >= 2` means the venue was found via multiple Wave 2
lenses (e.g., on the CMS sommelier list AND in a Vittles writeup AND in the
Heritage Foods customer list) — these are the highest-conviction leads in
Wave 2.
